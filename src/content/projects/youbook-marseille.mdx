A tourism office managing its activities across three different providers: every request risks triggering three API calls, three data formats, and three on-the-fly conversions. Youbook solves that. An aggregation layer that normalizes everything into **OCTO** and serves the data from an ElasticSearch cache, so the front-end never has to call a provider directly.

## Context

At Raccourci Agency, we build tourism booking solutions for our clients: tourism offices, sports federations, destinations. The common problem: their products are spread across providers like Regiondo, Yoplanning, or Welogin, and each one has its own format, its own API, its own constraints.

Youbook was born from that observation. Rather than letting the front-end call provider APIs directly, we centralize the aggregation, normalize according to the **OCTO** standard (the open standard for the tourism industry), and cache to absorb the load. On the interface side, **web components** allow deploying the same Youbook for any client by only changing the color palette.

I joined this project focusing on the ETL, the part that holds everything else together.

## How it works

The flow starts with **CRON jobs** that trigger import pipelines. For each client, we query their providers, fetch tourism products (activities, tickets, stays), and transform them into the OCTO format. The result is indexed into **ElasticSearch** to serve front-end queries.

This cache is central. Without it, every page navigation would trigger direct calls to the providers, with format conversions on top, and noticeable latency. With it, the front-end reads from ElasticSearch and providers are only queried during scheduled imports.

Microservices communicate via **Apache Pulsar**: each pipeline step publishes to a topic, and downstream services consume messages asynchronously. Each client has its own topics, which makes it possible to enable or disable a pipeline without affecting the others.

## Technical choices

**Kotlin + Ktor** for backend services: strong typing, coroutines for handling concurrent provider calls, and lightweight HTTP APIs without the overhead of Spring.

**JobRunr** for scheduling and monitoring import jobs. It handles job persistence, automatic retries, and provides a dashboard to track pipeline status without digging through logs.

**ElasticSearch** as the read layer: fast indexing of structured data, full-text search, filters by date, location, or availability. Its flexible schema also absorbs format variations when providers update their APIs.

**Web components** on the front-end: framework-agnostic, the components can be embedded in any CMS or static site. A new Youbook client means a new CSS theme, not a new codebase. Today, Youbook runs for several clients: Marseille Tourisme, the French Free Flight Federation, and more being deployed.

A payment module overhaul is underway to move away from provider widgets and take back control of the booking experience on Youbook's side.

## Challenges

**OCTO normalization.** Each provider has its own concepts: a "session" for one is a "timeslot" for another, availability is sometimes computed on the fly, sometimes pre-calculated. Mapping all of this reliably means as many edge cases as there are providers. The answer: one adapter per provider, with integration tests running against fixtures captured from the real APIs.

**Data freshness.** Products change constantly: prices, availability, descriptions. Finding the right balance between import frequency and provider load required instrumenting the pipelines to measure deltas between imports, and adjusting frequencies on a per-provider basis.

**Multi-client scalability.** Each client has different providers, different configurations, different languages. The ETL must be generic, but pipelines need fine-grained configurability. The Pulsar topic-per-client model isolates scopes well and prevents a failed import for one client from disrupting the others.

## What I took away

This project confronted me with the reality of production data: provider APIs are rarely 100% reliable, formats change without warning, and "simple" transformations become complex at scale. What taught me the most is the importance of observability on pipelines. Without metrics on what comes in, what goes out, and what fails, you're flying blind. Instrumenting before optimizing became a reflex.
